## What I Accomplished Today (06/06/2026):

**1. Fixed the Customer Account Page — Orders and Profile Now Display Correctly**

This was the main focus of the day. After logging in, customers were seeing "Good morning, there." (no name) and "No orders yet." even though real orders existed in the system. The admin dashboard was showing those same orders just fine, but the customer-facing account page was broken.

After detailed investigation and reading through the server logs on Vercel, three separate bugs were found and fixed — all in a single file (`app/api/auth/profile/route.ts`):

---

**Bug #1 — Shopify's Customer Account API Was Rejecting the Login Token**

The website was trying to fetch customer data using a new Shopify API called the "Customer Account API." This API requires a special token that starts with `shcat_` — but the token our login system produces is a completely different format and gets rejected every single time (401 Unauthorized error). This was causing a 133ms wasted request on every page load with a red error in the logs.

**Fix:** Removed the Customer Account API attempt entirely. It was never going to work with our current login flow, and there's no reason to keep trying it. The Admin API (see Bug #2) handles this job better.

---

**Bug #2 — The Admin API Query Was Using the Wrong Field Names**

After Bug #1 failed, the code fell back to querying Shopify's Admin API (the same one that powers the admin dashboard). But the query was structured incorrectly — it was trying to ask for a field called `financialStatus` inside a customer's orders, but that field doesn't exist in that context. Shopify returned a GraphQL error: *"Field 'financialStatus' doesn't exist on type 'Order'."*

This happened because an older version of the code was trying to load orders *nested inside* the customer query, which only exposes a limited set of fields. The correct approach is two separate queries:
- One query to fetch the customer's name, email, phone, and addresses
- A second query to fetch their orders by email directly from the orders list

Both queries were also updated to use the correct field names that the Admin API actually supports (`displayFulfillmentStatus` and `displayFinancialStatus` instead of the shorter versions), matching exactly what the working admin dashboard uses.

---

**Bug #3 — A Crash Was Causing a "500 Internal Server Error"**

After both Bugs #1 and #2 failed, the code tried one last fallback: decode the login token (a JWT) to extract the customer's email at minimum. But the code was trying to call `.split()` on a value that turned out to be a number, not a text string. In plain terms: it was like trying to cut a number in half with scissors — it doesn't make sense and crashes the program. This unguarded crash was causing the entire route to return a 500 error instead of failing gracefully.

**Fix:** Removed the risky JWT parsing entirely. Since the customer's email is already saved to their session when they log in (stored safely in an encrypted cookie), we just read it from there. No complex decoding needed. The entire handler was also wrapped in a try/catch so that if anything unexpected happens in the future, it fails gracefully instead of crashing with a 500.

---

**End Result:**

After deploying these fixes, the account page immediately started working:

- **Log line:** `[profile] Admin customer: jonathan.mauring17@gmail.com | orders: 6`
- **Account page shows:** "Good morning, John." with email displayed
- **Orders tab shows:** All 6 real orders with product images, names, prices, and dates
- **No more 500 errors or red logs**

The fix took the response from a 500 crash to a clean 200 success in 364ms — two Admin API calls running at the same time to fetch customer info and orders in parallel, same pattern as the admin dashboard.

4. **Built the full Promotional Email System** — A complete newsletter platform was built from scratch and is now live. Here is what was built across 9 code deployments:

   - **Database tables** — Two new tables were created in the database: one to store newsletter subscribers, and one to store email campaigns.

   - **Subscriber sign-up** — The "Subscribe" button in the website footer now saves the email address directly to the database. If someone unsubscribes and re-subscribes later, their record is cleanly reinstated. Duplicate sign-ups are handled silently with no errors shown.

   - **Unsubscribe link** — Every newsletter email contains a personal unsubscribe link. Clicking it opens a page that says "You've been unsubscribed" and immediately marks the person as inactive in the database.

   - **Email sending engine** — A behind-the-scenes function was built to send newsletters through the existing Resend email service. It sends in batches of 50 to stay within service limits, and includes a personalised unsubscribe link in every email footer.

   - **Admin marketing page** — A new "Marketing" page was added to the admin dashboard (visible in the sidebar between Communications and Reviews). It has two tabs:
     - *Subscribers tab* — shows how many active subscribers there are, displays their email and status, and has an Export CSV button to download the full list.
     - *Campaigns tab* — shows all past campaigns with their sent date and recipient count. A "+ New Campaign" button opens a compose panel where Scott can write a subject, body, optional button, and optional send date. There is a live preview so the email can be reviewed before sending. "Save Draft" saves it for later; "Send Now" sends it immediately.

   - **Weekly auto-send** — A scheduled task was set up to automatically send any newsletter that has been saved with a future date. It runs every Monday at 12:00 PM UTC (9 AM Atlantic Time). It checks for any draft campaigns that are scheduled for that time or earlier, sends them, and marks them as sent.

   - **Double-send protection** — If someone tries to send the same campaign twice (either by clicking Send Now a second time, or if the scheduled task fires at the same moment an admin sends manually), the system blocks the second send. Only one send ever goes out per campaign.

   - **Rate limiting on unsubscribe** — The unsubscribe link is public (no login required). To prevent someone from bulk-unsubscribing the entire list, the link is rate-limited to 10 clicks per minute per IP address.

5. **Generated and set the CRON_SECRET** — A secure random key was generated and added to both the local environment and the live Vercel server. This key is what allows the Monday auto-send to authenticate with the system.

## Discovered During Work:

- The admin overview page had `NEXT_PUBLIC_SUPABASE_URL` under the wrong name in the environment file — it was called `SUPABASE_URL` instead. Renamed and added to Vercel.
- The campaign send route and the weekly auto-send were both missing an error check after marking a campaign as sent — if the database write failed silently, the campaign would stay as "draft" and send again next week. Fixed with proper error handling.
- Both the admin Send Now and the auto-send cron were vulnerable to a race condition where two simultaneous sends could go out for the same campaign. Fixed with an atomic database lock that only one caller can claim.
- The newsletter subscriber opt-in was using a two-step check-then-insert approach that had a small window for duplicates. Replaced with a single atomic upsert.
- The Resend batch send was counting all emails as "sent" even if some were rejected by Resend. Fixed to only count confirmed deliveries.

### 9. Customer Communications Hub — Built and Deployed

This is the largest feature of the day. It replaces two Google Apps Script workarounds and brings customer communication into a single admin page at `/admin/communications`.

**Why this was built:** Contact form messages were going to an external Google script with no inbox. Bench notes (workshop reminders and notes about specific products) were also stored externally. Restock waitlists were buried on individual product pages and hard to act on.

**What was built:**

**Tab 1 — Restock Waitlist**
All products with people waiting for restock are now listed in one place, sorted by how many people are waiting. Each product shows the count of pending vs. notified signups, a collapsible list of email addresses, and a "Notify X waiting" button that sends the restock email blast immediately.

**Tab 2 — Contact Inbox**
Messages submitted through the store's contact form are now saved to the database and appear in this inbox. Each message shows the sender's name, subject, and arrival time. Clicking a message expands it to show the full text. Actions available: reply via email (opens a pre-addressed email draft), mark as read, mark as replied, and delete. An unread count badge appears on the nav item in the sidebar.

**Tab 3 — Bench Notes**
A simple notes board for workshop use. Notes can be added with an optional title, pinned to the top, edited, and deleted. Useful for things like "check the Hinks Globe before shipping — hinge is loose" or "waiting on glass shades for order #1043."

**Contact form change:** Previously, the contact form on the website sent data to a Google Apps Script URL. It now saves directly to the database and also sends an admin alert email via Resend. The form itself did not need any changes — only the backend was updated.

**New database tables:**
- `contact_messages` — stores all contact form submissions with read/replied timestamps
- `bench_notes` — stores workshop notes with pin and update tracking

Migration file: `docs/supabase/migrations/006_communications.sql` — must be run manually in the Supabase dashboard before these features go live.

### 10. Vercel Build Error Fixed — Lazy Database Client

After the Communications Hub was committed and deployed, Vercel's build failed with:

> `Error: supabaseUrl is required`

The cause: the database client was being created at the top of each API route file, which runs when Vercel scans files during the build process — before any environment variables are loaded. The fix was to wrap the client creation in a function (`getSupabase()`) that only runs when an actual request comes in. Applied to all six affected files.

Build passes cleanly after this fix.

## Discovered During Work

- Shopify blocks webhooks sent to `acmevintagesupply.com` (its own storefront domain). The webhook endpoint must use `acmelampandsign.vercel.app` instead. This is specific to Shopify's webhook system.
- Vercel evaluates module-level code in API route files during the build process. Any code that reads environment variables must be inside a function, not at the top of the file.
- The E2E cache sync delay (~5 minutes) for stock changes is expected behavior — not a bug. Shopify data is cached for up to 5 minutes to keep the storefront fast.
